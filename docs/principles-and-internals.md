## 1. 这份文档讲什么

这份文档不教你怎么启动项目，也不按文件逐个罗列接口。

它主要回答：

- 为什么当前系统要这样设计
- 这些设计分别在防什么问题
- 当前实现的关键取舍是什么
- 哪些点未来一定会继续演进

## 2. 为什么不能只做字符串裁剪

如果只做字符串裁剪，短期会很快，长期会遇到四个问题：

### 2.1 Token 统计不准

字符长度不是模型上下文长度。

结果就是：

- 预算判断不准
- 处理前后对比不准
- 实际收益不可比

所以当前系统把入口、处理中、输出都统一到真实 tokenizer 上。

### 2.2 不同内容类型被粗暴同化

代码、日志、表格、JSON、错误栈，压缩逻辑并不一样。

如果全走同一规则，常见后果是：

- 代码结构被破坏
- 错误栈信号丢失
- 表格信息被随机截断
- JSON 只剩乱码式片段

所以当前系统引入了 `RAGFragment` 类型。

### 2.3 语义损失不可追踪

如果只返回压缩后的字符串，而不记录过程，就无法回答：

- 到底删了什么
- 为什么删
- 是否还保留证据

这就是当前系统引入 `StepSemanticAudit` 的原因。

### 2.4 无法支持 page-out / page-in / 异步摘要

简单字符串裁剪只能在一次请求里完成，很难把大块内容：

- 暂存出去
- 延迟处理
- 再按需回填

所以项目需要 Redis 和异步摘要链路。

## 3. 为什么要用 Pipeline + Processor

### 3.1 这不是“为了架构而架构”

上下文治理天然是多步骤处理：

- 先分页
- 再去重
- 再微压缩
- 再结构化处理
- 再 snip
- 再异步摘要排队
- 最后 assemble

如果把这些逻辑全塞进一个函数，会出现：

- 难扩展
- 难调试
- 难观察每一步的收益
- 难做跳过策略

### 3.2 Processor 能力声明在解决什么问题

早期最容易犯的错是：

- 哪些步骤激进，写死在引擎里
- 哪些步骤只对结构化输入有效，靠 if-else 散落判断
- 哪些步骤要保 citation，没有统一表达

`ProcessorCapabilities` 的意义是把这些信息显式化。

于是引擎可以说：

- token 已经够小，就跳过 aggressive 步骤
- 没有结构化 chunk，就跳过 structured-only 步骤
- token 没到门槛，就跳过低收益步骤

### 3.3 为什么现在还要把实现继续下沉到 `core/components`

如果 processor 既负责“步骤编排”，又负责“具体文本 / RAG / prompt 规则实现”，目录虽然看起来分层了，但复杂度还是会重新堆回 processor。

所以当前代码又往前走了一步：

- processor 只保留步骤级编排和上下文胶水
- `TextSanitizer` 统一承接文本清洗顺序、规则和清洗报告
- `RAGNormalizer` 统一承接 RAG 排序、去抖和稳定化
- `PromptComponent` 统一承接 prompt section 组装与 stable prefix section 生成

这样做的价值不是“多一层抽象”，而是把高复用、高变化率的规则集中到一个可管理位置，避免：

- 同一条清洗规则散落在 `collapse / compact / canonicalize`
- prefix section 拼装逻辑残留在 `core` façade
- 新规则加入后只能在多个 processor 里重复修改

## 4. 为什么 `paging` 必须先存在

很多大模型系统的问题，不是“文本太长”，而是“某些 chunk 特别长”。

如果没有分页，只有两种粗暴选择：

- 全塞进 prompt，污染当前上下文
- 整块删除，损失信息

`paging` 提供第三种路径：

- 当前 prompt 只保留第一页
- 其余内容 page-out 到 Redis
- 后续需要时再 `PageIn`

这让系统开始具备“多级上下文治理”的雏形。

## 5. 为什么 `collapse` 要保留来源

简单去重很容易带来一个副作用：

- 内容虽然去重了
- 但引用来源被删没了

这在 RAG 场景中是危险的，因为证据来源本身就是上下文价值的一部分。

所以当前 `collapse` 做的是：

- 合并重复内容
- 保留并合并 `sources`

这是“压缩”和“证据保真”之间的平衡。

## 6. 为什么 `compact`、`snip`、`auto_compact_sync`、`auto_compact_async` 要分开

这四个步骤虽然都在“压”，但风险和成本不同。

### 6.1 `compact`

属于低风险清理：

- 清多余空行
- 不碰语义

### 6.2 `snip`

属于中等风险：

- 保头尾
- 删中间
- 适合超长高密度片段

### 6.3 `auto_compact_sync`

属于低延迟同步清洗：

- 主要针对日志、工具输出、错误栈
- 适合主请求内完成

### 6.4 `auto_compact_async`

属于高成本异步后处理：

- 需要排队
- 允许更重的压缩手段
- 不应阻塞主链

这四类逻辑如果不拆开，系统很快会变成“一个总压缩器”，难以控制延迟和风险。

## 7. 为什么当前 summary 仍然只是阶段性实现

当前 summary worker 已经是有效闭环：

- 能消费任务
- 能生成摘要
- 能写回 Redis
- `PageIn` 能优先返回摘要

但它仍然只是阶段性实现，原因有三层。

### 7.1 算法层还不稳定

当前是启发式规则，不是真实 LLM Provider。

### 7.2 存储层正在收敛，但还没走完

当前 summary 已经从“page key 的附属字符串”升级成结构化 `SummaryArtifact`，并带上：

- `artifact_id`
- `content_hash`
- `fragment_types`
- `provider / provider_version`
- `schema_version`
- `created_at / expires_at`

但它还没有完全演进成独立摘要子系统，当前仍然：

- 主要挂载在 page key 下读取
- 还没有 chunk 级独立索引
- 还没有多 provider 并存与切换策略

### 7.3 生命周期管理还不稳定

当前仍缺少：

- 真实外部 summary provider
- worker retry / claim 策略
- chunk 级独立摘要索引
- 更细粒度的摘要生命周期治理

所以当前 summary 的价值是“把架子搭起来了”，不是“最终方案已经确定了”。

## 8. 为什么当前最优先的不是继续加更多 Processor

表面上看，继续增加：

- `log_dedup`
- `tool_output_focus`
- `rag_rerank_trim`

似乎很自然。

但如果测试、观测、评测缺失，继续加 Processor 会有两个系统性风险：

### 8.1 你只能看到规则变多，看不到收益是否真实

### 8.2 一旦压缩伤害语义，很难定位是哪一步造成的

所以当前真正合理的顺序是：

1. 补测试
2. 补观测
3. 补评测
4. 再继续扩策略

## 9. 当前系统最重要的设计原则

### 9.1 Token 口径唯一

不能在不同地方混用：

- 字符长度
- rune 长度
- tokenizer 长度

### 9.2 Lossy 必须可解释

删了内容就必须留下审计。

### 9.3 结构化输入优先

不要再退回“everything is plain text”。

### 9.4 异步能力必须可降级

外部摘要模型故障时，主链不能跟着崩。

### 9.5 状态 key 必须可控且可复用

当前需要区分两类 key：

- 请求级元数据：适合带 `session_id / request_id`
- 可复用 artifact：应尽量用 content-addressed key

否则跨请求复用做不起来，prefix 命中收益也很难放大。

### 9.6 稳定内容必须尽量前置

prefix cache 的前提是“前缀逐 token 完全一致”。

所以应用层必须尽量做到：

- 稳定 RAG 在前
- 历史消息在中
- 当前活跃输入在后
- source / chunk 顺序稳定

这类设计不会替代下游推理引擎的 KV cache，但会直接决定它有没有命中机会。

## 10. 当前实现的边界和残余不确定性

### 已知边界

- 已有少量自动化测试，但还没形成覆盖 `service / store / worker` 主链的闭环
- 已有 Prometheus / Tracing / Grafana / Tempo 基线，但还没有 explain / replay / alerting 闭环
- 还没有离线 replay 评测工具
- 默认配置仍需要按环境填充，不能直接作为生产配置使用

### 残余不确定性

当前没有充分证据证明：

- 压缩率是否稳定达到目标 KPI
- 启发式摘要是否在真实业务数据上足够稳
- `TextSanitizer / RAGNormalizer / PromptComponent` 这轮组件化之后，处理顺序是否已经达到最佳收益
- 应用层 prefix cache 预测与下游真实 KV 命中之间的相关性是否稳定

这不是代码方向错误，而是“证据链还不够完整”。

## 11. 一句话理解当前系统

如果要用一句话讲清楚当前系统原理，可以这样说：

`它通过结构化输入、可编排处理器、可解释审计、可分页状态层和可异步摘要链路，把“上下文压缩”提升成“上下文治理”。`

## 12. 结论

这个项目最有价值的地方，不是某一个压缩规则，而是它已经开始把上下文处理从“零散技巧”变成“稳定系统”。

后续只要不偏离三个原则，方向就是对的：

1. 继续坚持结构化输入
2. 继续坚持审计可解释
3. 先补质量闭环，再补复杂能力
